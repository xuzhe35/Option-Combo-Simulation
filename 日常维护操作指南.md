# Option Combo Simulation 日常维护操作指南

## 结论：例行的市场数据维护只有 3 项

| 维护项目 | 建议频率 | 目的 |
|---|---:|---|
| Update Yield Curve | 每个使用日一次，建议在首次打开软件前 | 更新统一的 USD 折现曲线 `r(T)` / `D(T)` |
| Export implied λ | 每次建仓或重做 What If 前；盘中曲面明显变化后重新导出 | 保存当时、当品种、当期货月份的逐个周末/节假日 λ 曲线 |
| Update Exchange Calendars | 每周一次，建议周末执行 | 更新 NYSE、CME、NYMEX、COMEX 正式交易日历 |

启动/重启服务、安装依赖、清理日志和历史数据回填不属于每日市场数据维护。

---

## 1. Update Yield Curve

### 什么时候做

- 每个使用日第一次打开软件前执行一次。
- 如果早上更新时官方当日数据尚未发布，可在美国市场数据发布后再手动执行一次。
- 周末或节假日执行也是安全的：快照的 `curveAsOf` 会写成最近一个纽约营业日（周六/周日运行得到周五的日期），不会生成"未来日期"快照，What-If 会话仍会正常使用市场曲线。
- `ib_server.py` 在找不到最近营业日的曲线时会自动尝试更新一次，但不建议把自动补救当作常规操作。

### 怎么做

**Windows**

双击 [update_yield_curve.bat](update_yield_curve.bat)。

**macOS**

双击 [update_yield_curve_mac.command](update_yield_curve_mac.command)。

两个入口都会：

1. 读取 New York Fed SOFR 和 U.S. Treasury CMT 官方数据。
2. 生成统一的 USD `D(T)` 快照。
3. 将最新完整快照写入 `yield_curve/data/latest.json`。
4. 在终端中显示当前快照状态。

### 怎样算成功

终端最后应显示：

- `Yield-curve maintenance completed successfully.`
- 当前 `Curve as-of`
- `snapshotId`
- SOFR/Treasury 数据来源

软件中应保持 **Use unified daily SOFR / Treasury curve** 开启，状态不应是 `unavailable`。

### 失败时怎么做

- 更新失败不会覆盖上一份完整快照。
- 检查网络后重新双击执行。
- 如果软件显示正在使用可用的 cached curve，可暂时继续，但要注意 `as-of` 日期。
- 如果显示 `unavailable`，软件只能使用界面中的手动 `Fallback r%`；应先修复更新，再做精确 What If。

## 2. Export implied λ

### 什么时候做

implied λ 不是每天只变一次的参考利率，而是当时期权曲面的快照。建议在以下情况导出：

- 建立新组合前。
- 需要重新评估“从现在到目标平仓日”时。
- IV Term Structure 盘中明显变化后。
- 更换品种或 FOP 标的期货月份后。

### 导出前的条件

- `ib_server.py` 正在运行。
- TWS/IB Gateway 已连接，并有相关期权行情权限。
- Yield Curve 状态可用，或者你明确接受界面显示的 `Fallback r%`。
- 官方交易日历未过期。
- FOP 品种必须选择与 Portfolio 中相同的标的期货月份。

### 怎么做

1. 打开 [iv_term_structure.html](iv_term_structure.html)。
2. 点击 **Connect IB**，确认 IB 状态正常。
3. 在对应品种卡片中选择正确的期货月份（如果是 FOP），然后执行 Sync。
4. 等待页面明确显示 coherent 快照已经 **published** 到模拟器。同源页面会自动传递，不需要点 Export 才能交付；Export 用于归档、跨来源或跨机器。
5. 检查需要的目标日期已在 implied λ 覆盖范围内。如果后面的周末没有被覆盖，提高 **Option Streams** 数量并重新同步。
6. 点击该品种卡片的 **Export λ**。

文件名会包含品种、期货月份和 UTC 报价时刻，例如：

```text
implied_lambda_ES_202609_20260717T200102Z.json
```

### 在 Portfolio / Chart Lab 中使用

1. 打开 [index.html](index.html) 或 [chart_lab.html](chart_lab.html)。
2. 将 **Time Basis** 设为 **Weighted weekends (λ)**。
3. 保持默认已勾选的 **IVTS implied λ per weekend**；旧格式会话没有这个字段时也会自动迁移到严格模式。
4. 如果 IVTS 与 Portfolio 是同浏览器、同来源页面，最新合格曲线会自动传递；“同来源”要求协议、主机名和端口全部一致，`localhost:8000` 与 `127.0.0.1:8000` 不是同来源。否则点击 **Load λ File…** 载入刚刚导出的 JSON。
5. 核对状态中的品种、期货月份和市场日期，并确认明确显示 `coverage complete for N required non-trading dates`（所有必需非交易日已完整覆盖）。

### 重要提醒

- Export λ 文件保留计算时的市场报价时间作为审计信息；用户显式导入后，该冻结曲线不会因墙上时间过去而自动失效。
- 前端升级后，已打开的 IVTS、Portfolio 和 Chart Lab 页面都要各自强制刷新一次；只重启 `ib_server.py` 不会替换旧页签正在运行的 JavaScript。
- 若 IVTS 显示 coherent，而 Portfolio 仍显示 unavailable，按“两页强刷 → 核对完全同源 → 确认 localStorage 可写 → 重新 Sync”的顺序排查。
- 重新加载旧文件不会改写它的原始报价时间；选择文件本身就是用户对该冻结曲线的可用性确认。品种、期货月份和市场锚定日期仍必须匹配。
- Production Export 要求每个参与方程的到期都有 IB ContractDetails 的精确 `expiryAsOf`。λ 方程按交易所时区逐秒分段，期货还使用同一套 17:00 trade-date rollover；文件中的 `varianceTradingDays` / `varianceNonTradingDays` 是小数，因此不同到期 cutoff 相差一小时也不会被压成相同整数 DTE。
- 勾选 implied λ 后采用严格覆盖：只要任一“到目标时刻仍存续”的期权腿，从当前报价时刻到它的到期 cutoff 经过的周末或整日休市日不在 `byDate` 中，页面会红色列出缺失日期，Payoff 和 Probability 不再用标量值自动补齐。
- `not_required` 是唯一例外：没有任何到目标时刻仍存续的期权腿会跨过非交易日，因此此次计算本来就不需要 implied λ，不是在暗中使用标量补齐。
- 单一标量 λ、Calendar 和 Trading days 只保留作研究/显示。如果存续腿跨周末或整日休市，它们不能绕过 structured λ 安全阀；唯一豁免仍是明确的 `not_required`。
- FOP 按 `品种#标的期货月份` 严格匹配，例如 `ES#202609` 不能给 `ES#202612` 使用。如果目标时刻后仍存续、且需要 λ 的 FOP 腿绑定了多个期货月份，一条 V2 曲线无法同时满足；请统一绑定或分开评估。
- FOP 月份有两道校验：订阅时核对 `ContractDetails.underConId`，整面发布前再要求每条 option quote 的 `underlyingBindingVerified=true` 且实际月份一致。任一腿未验证或月份错误，整张 λ 快照会以 `ivts_option_contract_identity_invalid` 撤回，不会带着部分数据继续计算。
- Portfolio 的 Futures Pool 还有独立校验：必须是当前订阅代次、正确的 qualified `FUT` 合约，且 `conId/月份/乘数/交易所/币种` 和 120 秒新鲜度都通过。重新订阅、乱序旧代报价、错月或超时会先清空旧 Mark；在 Futures Pool 行和状态栏查看 `pending/rejected` 原因。
- SPX/NDX 的 Forward Carry 样本必须有 call/put/index 三方真实双边 BBO，并用 `ContractDetails.expiryAsOf - max(三方 quoteAsOf)` 的精确秒数计算。面板折叠不会停止刷新；交叉盘、单边盘、超时或到期身份不一致都会立即清掉旧 Carry。
- 如果同来源自动传递已经满足使用需求，“Export λ”只是为了归档、跨机器或跨来源使用；不需要重复导入。

### 目标日的精确时刻与到期 cutoff

界面选择的是目标“日期”，计算内部会把它解析成一个全组合共用的精确时刻：

- 目标是当前交易日、且没有未平仓腿在当日到期时，使用真实 `liveQuoteAsOf`。
- 目标是当前交易日、且近腿在当日到期时，cutoff 之前仍以近腿的精确 contract cutoff 为目标，因此“今天收官”不会被偷换成“当前报价时刻”；cutoff 通过后才改用 `liveQuoteAsOf`，避免对存续远腿倒推。
- 未来目标日有存续腿到期时，使用近腿唯一的 last-trade cutoff；远腿也在同一时刻估值。
- IB ContractDetails 的 `lastTradeDateOrContractMonth + lastTradeTime + timeZoneId` 优先，并转成 UTC `expiryAsOf`。`realExpirationDate` 可能更晚，不当作定价 cutoff。
- 如果同一目标日的多条未平仓腿拥有不同 cutoff，系统会以 `ambiguous_near_leg_cutoff` 停止，不取平均数。
- Live 模式默认强制合约级时刻：目标日到期的腿、目标后仍存续的 FOP/INDEX 腿、以及剩余不超过 7 天的任意存续腿，都必须有 IB 的精确 `expiryAsOf`。缺少时会红色列出 leg id 并以 `exact_contract_timing_missing` 停算。
- 合约身份和到期时刻通过独立的 metadata 消息交付，不依赖下一笔价格。即使 IVTS 已经订阅同一合约、Portfolio 只是复用 pooled ticker，也应在订阅挂接后立即收到；metadata 不会改写 BBO、IV、报价时钟或 Market Data Feed 健康状态。
- 只有完整可解析的 ContractDetails cutoff 才会缓存；不完整结果会在以后重新订阅时重试，同一 conId 的并发查询只发一次。当前版本若仍持续显示 `exact_contract_timing_missing`，应按真实 ContractDetails 不完整或合约身份校验失败排查，而不是关闭 IVTS 或改用宽松时钟。

IB 时间不可用时才使用品种档案的防御性备用时间：

| 品种家族 | 备用 cutoff |
|---|---|
| 股票/ETF、SPX、NDX | 16:00 美国东部时间 |
| ES、MES、NQ、MNQ | 15:00 美国中部时间 |
| CL | 13:30 美国中部时间 |
| GC、SI、HG | 12:30 美国中部时间 |

到达同时段结算的 cutoff 时近腿按内在价处理；当日仍在 cutoff 之前时，0DTE 腿保留剩余的小时数。这些备用时间不能替代具体合约的 IB 数据。

例外：标准月度 SPX 和传统季度 AM 结算的 ES/NQ/MES/MNQ，最后交易时刻之后还要等 SET/SOQ 开盘特别结算值。该值不是最后交易时刻的盘面指数/期货价，因此目前会以 `deferred_settlement_fixing_unsupported` 停算，不画伪确定的内在价曲线。周度、EOM 和 PM 结算合约不受影响。

### 推荐操作顺序

1. 先在 Portfolio 的 Futures Pool 绑定好每条 FOP 腿的真实标的期货月份。
2. 再在 IVTS 选择同一月份，Sync 足够多的成对到期，直到覆盖最远的存续腿。
3. 等待 coherent snapshot，保持 IVTS 连接，然后勾选 implied λ。状态只有绿色 `coverage complete ...` 或明确的 `not_required` 时才继续。
4. 设置目标平仓日；如果它是近腿到期日，无需手动猜收盘小时。等待 `Pricing target` 显示 IB near-leg cutoff；如果列出缺失 leg id，先确认已运行当前后端版本、TWS 已连接，再重新订阅。IVTS 可以保持打开，不需要用关闭它来规避订阅复用。
5. 组合的 Live Price 保持默认 **Midpoint**，并确认每条远腿都有新鲜有效的双边 BBO。此时软件会用自己的 BSM/Black-76、quote 时刻的 Forward/r 和精确 λ 时钟反解逐腿本地 IV；腿部会显示 `Local BBO`。Portfolio Mark、model、last、manual 和单边盘口不会用于这一步。
6. 对 FOP 再确认 Futures Pool 显示当前代次身份已验证，而不是 `pending/rejected`；对 SPX/NDX 确认 Forward Carry 样本不是 `stale/unavailable`。
7. 只要 λ 状态变红、Futures/Carry 身份失效，或腿部显示 `Local BBO` 校准失败，就先恢复报价、重新 Sync、增加 Option Streams 或校正月份，不要把空图当成零收益。取消 implied λ 或改成 Calendar/Trading/标量不会绕过跨非交易日的安全阀。

本地 BBO IV 的作用是消除“同一个当前价格被不同模型输入重算后对不上”的基差；它仍然是假设未来 smile/skew、流动性和相应 Forward 情景不变的条件预测。Live What-If 默认使用 `strict-bbo`：目标时刻还存续的每条期权腿都必须完成本地 BBO IV 反解，否则 Valuation、Payoff、Chart Lab、Probability 和 Amortized 全部停算。目标时刻已到期、只算内在价的近腿不要求 BBO。选择 **Portfolio Mark** 仍可用于 Live P&L，但它不能冒充远腿的严格 What-If 锚点。

严格本地反解要求 option BBO、Forward/spot 和全组合 live clock 的时间差不超过 **30 秒**（普通行情的新鲜度上限仍为 120 秒）。这是为了防止只剩几分钟的腿用异步价格反解出假 IV。如果同时存在 structured λ 缺口与 BBO 失败，界面会优先报告 `implied_lambda_incomplete` 并列出缺失的周末/节假日，不会用泛化 BBO 错误遮盖根因。

WebSocket 断开时会立即使 strict projection 失效并重画；另有每 5 秒检查的 watchdog，连续 120 秒没有收到市场行情 payload 会标记 `feed_stale`。旧报价可以继续显示在 Live P&L 作为诊断，但会标为 stale，不能继续画 What-If。

Chart Lab 自己的第二路 WebSocket 只服务日 K 和画面上的即时价格线。组合定价始终使用 Portfolio 主 WebSocket 的整套原子状态，不会把第二路价格和主路的 BBO 时刻、Forward/Carry、Futures Pool 或利率曲线拼在一起。

历史回放不受这个 live gate 影响。旧的“直接使用输入 IV”路径仅保留为 JSON 会话中显式的 `projectionConvergenceMode: "legacy-input-iv"` 兼容值；新会话、无此字段或非法值都是 `strict-bbo`。

IVTS 的自动历史采样是独立的研究功能：从上次成功样本起经过 60 分钟才到期，不是每个整点强制写入。这些小时样本不会延长实时 V2 曲线的 120 秒新鲜度。

## 3. Update Exchange Calendars

### 什么时候做

- 每周一次，建议在周末执行。
- 如果 IVTS 显示 `calendar_unavailable`、日历过期或覆盖不足，立即额外执行一次。
- 浏览器会把超过 14 天未更新的正式日历视为不可用，不会使用节假日公式猜测。

### 首次配置 CME 凭据

如果使用 ES、NQ、MES、MNQ、CL、GC、SI 或 HG，需要在 `config.local.ini` 的 `[cme]` 部分填写：

```ini
[cme]
api_id = 你的_CME_API_ID
api_secret = 你的_CME_API_SECRET
```

也可使用短期 `access_token`。`config.local.ini` 已被 Git 忽略，不要将真实凭据提交到仓库。

### 怎么做

**Windows**

双击 [sync_exchange_calendars.bat](sync_exchange_calendars.bat)。

**macOS**

双击 [sync_exchange_calendars_mac.command](sync_exchange_calendars_mac.command)。

成功后会同时更新：

- `exchange_calendars/official_exchange_calendars.json`
- `js/official_exchange_calendars.generated.js`

这两份文件是同一份日历的源数据和浏览器版本，不应手动分别编辑。

### 怎样算成功

- 脚本返回成功，无非零退出码。
- 生成文件中的 `fetchedAt` 是本周时间。
- 包含实际使用品种的 calendar，例如 `CME:ES`、`NYMEX:CL`或 `COMEX:GC`。
- 重新打开 IVTS 后不再显示 `calendar_unavailable`。

### 失败时怎么做

- 同步脚本在下载或解析失败时不会写入半成品。
- 检查 CME API 凭据、网络和终端错误后重试。
- `--nyse-only` / `-NyseOnly` 只能作为 NYSE 启动用途；如果要使用 FOP，不能用 NYSE-only 代替 CME/NYMEX/COMEX 更新。

## 不属于日常市场数据维护的项目

以下工作只在有需要时执行，不需要每天做：

- 启动或重启 `ib_server.py`。
- 安装/更新 Python 依赖。
- 清理运行日志和 pid 文件。
- 历史数据回填。
- 旧 `rates.db` Treasury 导入。
- 订阅 SOFR 期货并反推利率。当前日常 Yield Curve 不需要任何 SOFR 期货行情订阅。
- IVTS 历史样本整理或研究回测。
